## B5 — 强化测试 / 行为 eval

- **目标**：把"测 CLI"升级为"也测 harness/skill 的行为"，给后续迭代兜底。
- **依据**：Superpowers 用 drill eval harness 测 skill 行为，而非只单测代码。
- **改法**：
  - 补单测：`init`/`upgrade` 全文件集 idempotency（连跑两次零变更）；`upgrade` 幂等；`check` 的 layering 命中分支；`migrate` advisory 分支。
  - 新增 `evals/`：声明式「场景 → 期望」清单（如「目标库已有旧 current-status.md」→「migrate 提示 rename 且不覆盖」），先做一个轻量 runner（纯标准库），不引依赖。
  - 可选：skill 触发 eval（给定 prompt 描述，断言 description 关键词匹配）——先文档化人工 eval 清单，避免过度工程。
- **验收**：测试数显著上升且全绿；`evals/` 有可运行 runner + README。
- **风险**：行为 eval 易过度工程 → 先覆盖高价值场景，YAGNI。

## B6 — memory 生命周期

- **目标**：防止 `.harness/memory/lessons.md` 变成新的无界启动文档（即本项目立项要解决的问题在 memory 维度复发）。
- **依据**：Anthropic structured note-taking 强调 memory 需要 curation；母本 memory.md 已在增长。
- **改法**：
  - `check` 增 warning：`lessons.md` 超过阈值（如 200 行）提示 review/归档（阈值进 state.yml limits）。
  - 文档化「晋升/淘汰」约定（长期 lesson 留 lessons，一次性细节留 phase archive）——D4 已起头，B6 补 check 兜底。
  - 可选 `harness memory prune`（把陈旧条目移到 `.harness/memory/topics/archive`）——先 YAGNI，仅当 warning 不够再做。
- **验收**：超阈值 lessons 触发 check warning；约定写入 lessons 模板。
- **风险**：阈值误报 → 设为 warning 非 error，可配置。

## C4 — 可选 slash commands

- **目标**：对齐 GSD「command 即工作流」，让 phase 操作有顺手的入口。
- **依据**：GSD 的 `/gsd-*` 命令；CC 的 `.claude/commands/`。
- **改法**：
  - `init`/`upgrade` 加 `--with-commands`（默认关）：生成 `.claude/commands/harness-*.md`（如 `harness-phase.md`、`harness-recall.md`），每个是引导对应 CLI/phase 流程的 prompt 模板。
  - 与 C2 hook 同样幂等、可回退、不强加。
- **验收**：`--with-commands` 生成 command 文件；默认不生成；幂等。
- **风险**：command 与 skill 内容重叠 → command 只做「动作入口」，知识仍在 skill/POLICIES。

## D5 — out-of-scope 一等产物

- **目标**：把「明确不做」从单个 PLAN 升到项目级，防止同一问题反复被问。
- **依据**：GSD `.out-of-scope/` 一等公民；CC self-contained spec 要写 out-of-scope。
- **改法**：
  - 新增 `docs/ai-harness/OUT-OF-SCOPE.md` 模板（项目级明确不做 + 原因 + 触发重审条件），INDEX 引用。
  - 与 PLAN 的 phase 级 Out of Scope（D1）分层：项目级稳定、phase 级临时。
- **验收**：init 生成 OUT-OF-SCOPE.md；INDEX 引用且 check 0 error。
- **风险**：与 DECISIONS 重叠 → OUT-OF-SCOPE 只记「不做什么」，DECISIONS 记「决定怎么做」。

## D6 — 收敛 current/ 文件冗余

- **目标**：消除 `checkpoint` 同时写 EVIDENCE/HANDOFF/PROGRESS 三处近似内容的冗余。
- **依据**：12-factor #5 unify execution & business state；冗余本身制造上下文膨胀与发散。
- **改法**：重新划分职责——
  - `PROGRESS.yml` = 机器可读 checkpoint 列表（time/status/note）。
  - `EVIDENCE.md` = 人读验证证据 + Verify Checklist。
  - `HANDOFF.md` = **只在 compact 时**由骨架生成 + agent 填（恢复用），checkpoint 不再追加 HANDOFF。
  - 改 `phase_checkpoint`：去掉对 HANDOFF 的 append。
- **验收**：checkpoint 后 HANDOFF 不被污染；compact 才产出 HANDOFF；现有测试相应更新仍全绿。
- **风险**：行为变更可能影响依赖旧流程的人 → 在 VERSIONING 记录，必要时 bump schemaVersion。
