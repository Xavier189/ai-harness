# ADR-0001：红线重定义——单文件从「红线」降级为「默认 guardrail + 逃生口」

## 状态

已接受。取代 HD-0002 中"单文件"部分（零依赖部分保留，见 HD-0002'）。

## 背景

- 早期四条红线：单文件 / 零依赖 / 可选可回退 / 不增启动负担。其中「单文件」(`ai_harness.py`) 在 v0.1 承担核心价值=「copy 一个文件进任意仓库就能跑 / 易审计」。
- v0.7 引入 `pyproject + console_scripts + uvx/pipx` 后，**分发已由打包方案接管**：用户 `pip/uvx` 装完，感知不到底层是 1 个还是 N 个文件。单文件对「分发」不再 load-bearing。
- 反效果显现：单文件开始成为**拒绝有价值功能的理由**（G1 git 集成被判"最易侵蚀单文件红线"而挂起）。这是"约束在守护身份"还是"cargo cult"的典型分叉点。

判定红线的测试（三问，任一为否则非红线）：
1. 是**用户可观测属性**，还是内部实现细节？
2. 违反后**产品身份/价值变**，还是只是内部结构变？
3. 守护的是**不可逆 / 难收回的伤害**吗？（红线守单向门）

四条过测试：

| 红线 | 用户可观测 | 身份变 | 守不可逆伤害 | 判定 |
|---|---|---|---|---|
| 零依赖 | 是 | 是 | 是（供应链门一旦开难收回） | ✅ 真红线 |
| 不增启动负担 | 是（harness 核心卖点=省 context） | 是 | 部分 | ✅ 真红线 |
| 可选可回退 | 是（旧库不被破坏） | 是 | 是（破兼容伤信任） | ✅ 真红线 |
| 单文件 | **否** | **否**（拆成小 package 仍零依赖、仍一个 `harness` 命令，用户零感知） | 否（随时可逆） | ❌ proxy |

## 决策

1. **真红线收敛为三条**：零依赖、不增启动负担、可选可回退。新增一条更本质的补位红线：**「装/用零摩擦」**（一行装、一个命令、零配置）——这才是单文件当初真正想保护、且用户可观测的东西。
2. **单文件降级为「默认 guardrail」**：能单文件就单文件（白嫖它对膨胀的最小化压力，天然 YAGNI 绊线）。
3. **给逃生口**：当某功能确实需要时，允许拆成小 package（`ai_harness/` + 少数模块）。放行闸门 = 三条真红线全过 **且** 满足新锚「拆完 import 结构仍能被一个人一口气读懂」。
4. 单文件本身**不得再作为拒绝功能的独立理由**；评估新功能时，用三条真红线过闸，不用"文件数"过闸。

## 结果

- 正面：解锁演进（G1 等按真闸门重评，见 backlog C）；红线从"内部实现细节"转为"用户可观测不变量"，更抗漂移。
- 负面/风险：失去单文件这条"便宜显眼的防膨胀绊线" → 有 sprawl 风险。缓解=新锚「一个人能一口气读懂」+ 保持单文件为默认，拆分需显式过闸并记 ADR。

## 约束

- 拆分不是默认动作：必须由具体功能需求触发、显式过闸、留 ADR，不得顺手重构式拆。
- 零依赖红线**不随本 ADR 松动**：拆成 package 也只能用 Python 标准库。
- 启动负担红线不松动：拆分不得增加每次 session 冷启动的加载/阅读量。
